#disk #theory #partitions

## **What is Partitioning?**

A [[01. Introduction to Disk|disk]] by itself is just raw space — partitioning is how you divide that space into separate sections, each with its own purpose.

Think of it like a house. The disk is the land, and partitions are the rooms. Each room serves a different purpose — bedroom, kitchen, storage. Linux works the same way.

---
## **Why partition?** 

You separate concerns. For example, if `/var` fills up completely (logs going crazy), it won't crash your whole system because it's isolated in its own partition. Same idea for `/home`, `/boot`, etc. It also makes reinstalling the OS easier — you can wipe `/` without touching `/home`.

**What happens technically?** When you partition, you're just writing a table (MBR or GPT) to the disk that says "this chunk of space starts here and ends here." No formatting happens yet — that comes when you create a filesystem on the partition.

***The flow is always:*** *Disk → Partition → Filesystem → Mount → Use*

---

## **MBR and GPT**

There are two main parititioning method: **MBR** and **GPT**.

**MBR (Master Boot Record)** is the old standard. It has a hard limit of 4 primary partitions and only supports disks up to 2TB. It stores partition info in the first sector of the disk. Still exists because of legacy systems.

**GPT (GUID Partition Table)** is the modern replacement. It supports up to 128 partitions, handles disks up to 8 zettabytes (ZB), and is much more reliable because it stores partition info in multiple places on the disk (so if one gets corrupted, it can recover). **GPT** scratches a disk more deeply than **MBR**, so it is easier to repair the data saved via **GPT**.

**Which one do you use?** In RHEL 9 and any modern system, GPT is the default and the right choice. MBR is only relevant if you're dealing with old hardware or legacy VMs.

**The tool connection:** `fdisk` traditionally works with MBR, `gdisk` works with GPT. However, modern `fdisk` on RHEL 9 also supports GPT — but `gdisk` is cleaner for GPT work. `parted` handles both.

---

## **What is Extended Partition?**

This is an MBR-specific concept — doesn't exist in GPT.

**The problem:** MBR only allows 4 primary partitions. But what if you need more than 4?

**The solution they invented:** make one of those 4 slots a special "container" called an **extended partition**. Inside that container you can create unlimited logical partitions.

So the structure looks like this — you have up to 3 primary partitions, then 1 extended partition which holds your logical partitions inside it.

**Numbering in Linux:** primary partitions take numbers 1-4. Logical partitions always start from number 5, even if you only have 1 logical partition. So `/dev/sda5` is always the first logical partition regardless.

**Extended partition itself** holds no data — it's just a wrapper. You can't format or mount it. Only the logical partitions inside it are usable.

**Why does this matter today?** Mostly history. Since GPT supports 128 partitions natively with no tricks needed, extended partitions are irrelevant on modern systems.

## **How to see all disks and their partitions?**

It is very easy. Just run `lsblk` command to see them.
Example:

```bash
NAME          MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
sr0            11:0    1 12.7G  0 rom
nvme0n1       259:0    0   10G  0 disk
├─nvme0n1p1   259:1    0  600M  0 part /boot/efi
├─nvme0n1p2   259:2    0    1G  0 part /boot
└─nvme0n1p3   259:3    0  8.4G  0 part
  └─rhel-root 253:0    0  8.4G  0 lvm  /
nvme0n2       259:4    0    5G  0 disk
```
Such as, `nvme0n1` is a disk (SSD) and `nvme0n1p1`, `nvme0n1p2`, `nvme0n1p3` are its partitions.
