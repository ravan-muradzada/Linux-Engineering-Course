## **Intro to SUID**

SUID (Set User ID) is a special permission bit that applies **only to executable files**. When SUID is set, any user who runs that program executes it with the **effective user ID (EUID) of the file owner**, not their own user ID. In most real cases, the owner is `root`, so the program runs with root privileges temporarily.

How it works internally:  
When you execute a normal program, the process runs with:

- Real UID (RUID) → who started it
- Effective UID (EUID) → whose permissions are used for access checks

---

## **Usage**

With SUID enabled, the kernel sets the **EUID to the file owner**. That means permission checks are done using the owner's privileges. After the program finishes, everything returns to normal. This is controlled by the kernel — not by the shell.

Where it is used:  
SUID is used for programs that need temporary elevated privileges to perform restricted operations. The classic example is:

passwd

Normal users cannot modify `/etc/shadow` directly because it is writable only by root. But `passwd` is owned by root and has the SUID bit set. So when you run it, it temporarily runs as root, updates the password safely, and exits.

Another common example is:

sudo

It also relies on controlled privilege elevation.

You can check SUID files with:

```
find / -perm -4000 2>/dev/null
```

How to set it:

- Symbolic: `chmod u+s file`
- Numeric: `chmod 4755 file`  
    (4 = SUID)

---

## **s and S**

Now the important visual difference: `s` vs `S`.

When you run `ls -l`, look at the owner's execute position:

- `rwsr-xr-x` → lowercase **s**  
    This means SUID is set **and execute permission is also set**. The program is executable and SUID is active. This is correct and functional.
- `rwSr-xr-x` → uppercase **S**  
    This means SUID is set **but execute permission is NOT set**. The file is not executable. So even though SUID exists, it does nothing because the program cannot be executed.

So:

- `s` = SUID + executable (working state)
- `S` = SUID without execute (misconfigured or useless state)

Important security warning:  
SUID on root-owned binaries is powerful and dangerous. If a SUID-root program has a vulnerability (buffer overflow, improper input handling, etc.), it can be used for privilege escalation. That’s why modern systems keep the number of SUID programs minimal.

Also note:

- SUID does not work on scripts in most modern Linux systems (for security reasons).
- It is ignored on directories.
- It is mainly used for compiled binaries.
