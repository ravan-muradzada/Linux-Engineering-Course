## **Label and Context in SELinux**

**Label and Context are essentially the same thing.** The label _is_ the context. "Label" is just the general term for the tag attached to something, and "context" is the actual value of that label. Think of it like: label = the concept, context = the data.

---
## **Viewing Context**

When we run `ls -Z`, we can see the context of the files:

```
system_u:object_r:httpd_sys_content_t:s0
```

The third property here, **type** (`httpd_sys_content_t`) is the one that actually matters. 99% of [[01. Introduction to SELinux|SELinux]] decisions are made based on this field alone.

Let me walk through the exact decision SELinux makes, step by step.

---
## **Steps**

##### Allowed
When a process tries to access a file, SELinux asks one question: **does a rule exist that allows this source type to access this target type?**

Take this real example — Apache trying to read a file:

- Apache process context type: `httpd_t`
- File context type: `httpd_sys_content_t`

SELinux looks into the policy and finds a rule that says: _"httpd_t is allowed to read httpd_sys_content_t"_ → **allowed.**

##### Denied
Now imagine you created a file in root directory. The file content is fine, Apache is running fine — but the file still carries its original label from root directory: `default_t`

SELinux checks: _does a rule exist allowing httpd_t to read user_home_t?_ → **No rule found → blocked.** Apache returns a permission denied error even though Linux rwx [[01. Introduction to Permissions|permissions]] are perfectly fine.

This is the most common real-world SELinux problem and exactly why context type is everything.

---
## **Changing Context Type** 

If we want to allow denied processes, we need to change the context type of the file created by us.

There are two command to help us in this case. However, they work differently, and it is important to choose one wisely.

### **`chcon`**
**chcon** changes the context directly on the file, but the policy database doesn't know about it. So if you run `restorecon` later, or the filesystem gets relabeled on reboot, your change gets wiped and the file goes back to whatever the policy says it should be. It's temporary in nature — useful for quick testing, not for production fixes.

```bash
chcon -t httpd_sys_content_t /var/www/html/myfile.html
```

### **`semanage fcontext`**

**semanage fcontext** changes the policy database — it tells SELinux "files at this path _should_ have this type." But it doesn't immediately relabel existing files. You then run `restorecon` to apply it.

```bash
semanage fcontext -a -t httpd_sys_content_t "/var/www/html(/.*)?"
restorecon -Rv /var/www/html/
```

This combination is permanent. Even after reboot or relabeling, SELinux checks the policy database, sees your rule, and keeps the correct label.

`restorecon` reads the policy database — the rules set by `semanage fcontext` — and applies the correct context to files that don't match what the policy expects.


> [!Info] `semanage-fcontext` command
> It is not important to learn these commands by heart. We can just use `man` to see examples about them. We just run `man semanage-fcontext`. We find these lines: 
> ```
> EXAMPLE
> Remember to run restorecon after you set the file context
> Add file-context httpd_sys_content_t for everything under /web
> # semanage fcontext -a -t httpd_sys_content_t "/web(/.*)?"
> # restorecon -R -v /web
> ```
> We need to take the command from there. The only thing to change it the directory path. We should write the path into `/web` part and not touch to `(..)`.

## **Ports for SELinux**

Just like files need the right context type for SELinux to allow access, **[[04. Protocols and Ports|ports]] do too.** `semanage port` manages the SELinux labels assigned to network ports.

A real example: say you change [[02. Apache|Apache]]'s listening port from 80 to 8090 in the config. Apache fails to start — not because of firewall or config issues, but because SELinux only knows Apache
(`httpd_t`) is allowed to bind to ports labeled `http_port_t`, and port 8090 doesn't have that label by default.

We can solve the problem in the similar manner:

```bash
semanage port -a -t http_port_t -p tcp 8090
```

We can check it with this command afterward: `semanage port -l`.

We do not need to use `restorecon` for ports, it is for files and directories on the filesystem.
