## **Image**

An image is essentially a **blueprint** for a container.

It's a read-only file that contains everything needed to run an application — the OS base layer, libraries, config, and the app itself. When you _run_ an image, it becomes a live container.

A simple analogy: if a container is a running process, the image is the executable file behind it. Or think of it like a class and object in OOP — the image is the class, the container is the instance.

Images are built in **layers**. Each instruction (install this package, copy this file) adds a layer on top of the previous one. This makes images efficient — layers are cached and shared between images where possible.

In practice, you either pull a ready-made image (like `rhel`, `nginx`, `postgres`) from a registry such as **Docker Hub** or **[[02. Introduction to Red Hat|Red Hat]]'s registry**, or you build your own using a `Dockerfile` (Docker) / `Containerfile` (Podman).

One important thing to internalize early: **the image itself never changes**. You run it, do things inside the container, but the original image stays untouched. If you want a modified image, you build a new one.

## **Installing and Running Image**

**`podman pull`** downloads an image from a registry to your local machine. That's it — no container is created yet, you just have the image stored locally.

```bash
podman pull nginx
```

**`podman run`** **creates a container** from an image and starts it. If the image doesn't exist locally, Podman automatically pulls it first — so in many cases you can skip `pull` entirely and just use `run`.

```bash
podman run nginx
```

The key distinction: `pull` is just downloading, `run` is actually doing something with it.

Without any options, `run`:
- Gives random name to container
- Takes full terminal space (foreground process)

We can see installed images using:

```bash
podman images
```


> [!Warning] Process in Container
> The thing that makes container run is a [[01. Introduction to Process|process]] inside it. If a process inside the container dies the container will fail and die. 
> It is recommended to put only one process inside the container. (Default one)

## **Creating and Deleting Container**

### Creating Container
As I mentioned above we use `run` command to create a container. Creating container is basically both installing an image(if not exists) and run it.

We create container like that:

```bash
podman run -d --name <CONTAINER_NAME> <IMAGE_NAME>
```

- `-d` runs the container in the background (detached mode), so your terminal isn't locked.
- `--name <NAME>` gives the container a name instead of a random one.

### Viewing Container
We can look at our existing podman containers using `ps` command:

```bash
podman ps
podman ps -a
```

- `podman ps` - Shows running containers
- `podman ps -a` - Shows all containers including stopped ones.

### Starting and Stopping Container
We use `start` and `stop` to start or stop the containers:

```bash
podman start <CONTAINER_NAME>
podman stop <CONTAINER_NAME>
#Instead of Container Name, we can also use Container ID (or its the first 2 characters)
```

### Deleting Container
We use this command to remove a container completely:
```bash
podman rm <CONTAINER_NAME>
#Instead of Container Name, we can also use Container ID (or its the first 2 characters)
```

We need to stop the container before removing it.

### Deleting Image
We use this command to remove an image from the system:
```bash
podman rmi <IMAGE_NAME>
```

We need to remove containers on it, before removing image.

### Entering Container Bash
Most containers work with Linux distros. We can enter them and work inside. To be able to enter a running container, we use this command:

```bash
podman exec -it <CONTAINER_NAME> /bin/bash
```

## **Mounting in Containers**

A container is isolated and **temporary**. Any files created inside it disappear when the container is removed. Mounting solves this by connecting a directory from your **host machine** into the container.

```bash
podman run -v /host-dir:/container-dir:Z nginx
```

Whatever exists in `/host-dir` becomes visible inside the container at `/container-dir`. Changes made inside the container reflect on the host, and vice versa. The data lives on the **host**, not inside the container.

**The core idea to internalize:** containers are meant to be disposable. Your application code and data should live outside the container on the host (or in a dedicated volume), and the container just uses it. This way you can destroy and recreate containers freely without losing anything.


> [!Warning] SELinux Assurance
> We should add `:Z` at the end of mouting. Adding `:Z` tells Podman to automatically relabel that directory so SELinux allows the container to access it.

## **Port Forwarding**

When a container runs, it's isolated — including its [[01. IP Address and Its Variations|network]]. The container has its own internal ports, but your host machine can't reach them directly. Port forwarding (also called port mapping) creates a **bridge** between a host port and a container port.

```bash
podman run -p 8080:80 nginx
```

This means: any traffic hitting **port 8080 on your host** gets forwarded into **port 80 inside the container**. Nginx inside the container is listening on 80, but you access it from outside via 8080.

We will access port 8080 of VM and we will be redirected to port 80 of the container from here.
We need to add 8080 to [[04. Firewall|firewall]] in VM.

## **Creating Container with Full Config**

We can create the container like that:

```bash
podman run -d --name <CONTAINER_NAME> -v /HOST-DIR:/CONTAINER-DIR:Z -p <HOST_PORT>:<CONTAINER_PORT> <IMAGE_NAME>
```
