## **How to set new crontab?**

First, we should know that each user has its own [[01. Introduction to Crontab|crontab]] list. There just one file for each and they add all their crontabs to this file.

We use this command to open crontab file:

```bash
crontab -e
```

Root user can create a crontab for any user in the system. It just need to run this command:

```bash
crontab -e -u <USER_NAME>
```

After running this command, the user himself can see the crontab created by root in this crontab list.

If we just want to see what is in our crontab file, we run:

```bash
crontab -l
```

## **Where Are All Crontab Files Saved?**

If we go to `/var/spool/cron` file, we can see all crontab files created by users in the system:

```bash
cd /var/spool/cron
```

We can delete any user's crontab file by running:

```bash
rm -f <USER_NAME>
#crontab files are named by their owner's name
```

## **Cron Directories**

There are cron related directories under `/etc`: 
`cron.d/`,`cron.daily/`, `cron.hourly/`  `cron.monthly/` `cron.weekly/`.

These directories are a clean way to manage scheduled tasks without touching a single crontab file. When you need a script to run on a predefined schedule, you simply drop an executable script into the matching directory — `/etc/cron.hourly`, `/etc/cron.daily`, `/etc/cron.weekly`, or `/etc/cron.monthly` — and the system handles the rest. For example, placing a `cleanup.sh` script in `/etc/cron.daily` will run it once a day, no timing fields needed, just make sure it has execute permission with `chmod +x`. 

The `/etc/cron.d` directory works differently — it accepts files with full crontab syntax including the timing fields, and importantly requires a username field like `30 2 * * * root /usr/local/bin/backup.sh`, because unlike personal crontabs the system needs to know which user runs the job. 

Packages like `logrotate` or `sysstat` use `/etc/cron.d` to install their own schedules without interfering with anything else. The actual execution of `cron.daily`, `cron.weekly`, and `cron.monthly` is handled by **anacron**, which means if the system was off at the scheduled time, it will still catch up when it boots — with the built-in delays (5, 25, 45 minutes). To see these delays, we need to run `cat /etc/anacrontab`.

## **Important Tip**

> [!Warning] Important Tip About Crontab
> System does not run crontab file line by line. It runs all lines together.
