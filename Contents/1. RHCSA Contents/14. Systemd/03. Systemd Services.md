#systemd #services

## **What is Service?**

Services in **systemd** is used to run the programs in the background. Every service in systemd is defined by a **[[02. Units|unit]] file** — a plain text config file that tells systemd _what_ to run and _how_. Service unit files end in `.service`. 

---

## **Syntax of Service File**

As an example of service file, it is very good to show `sshd` service:

```bash
cat /usr/lib/systemd/system/sshd.service
```

A typical service file has three sections: `[Unit]`, `[Service]`, and `[Install]`.

#### **`[Unit]`**

- `Description=` — just a human-readable label, shows up in `systemctl status`.

- `After=network.target` — don't start this service until networking is up. This is about _ordering_, not dependency. It doesn't _require_ network, just waits for it. `After=x` means "Run this only after `x`".

- `Requires=` — if this unit fails, mine fails too. Hard dependency.

- `Wants=` — softer version of Requires. If it fails, my service still tries to start.

---

#### **`[Service]`**

- `Type=` — how systemd tracks the process. The most common ones are `simple` (default, process starts and that's it) and `forking` (old-style daemons that fork into background).

- `ExecStart=` — the actual command that runs the daemon. This is the most critical line.

- `ExecStop=` — command to stop it cleanly. Optional; systemd sends SIGTERM by default.

- `Restart=on-failure` — if the process crashes, restart it automatically. Very useful.

- `User=` — run the process as this user instead of root.


> [!Info] Types under `[Service]`
> In the systemd service file under the `[Service]` there is `Type` section. The most important options for this is them:
> - **simple** — the default. systemd starts `ExecStart=` and immediately considers the service running. The process stays in the foreground. Most modern daemons work this way.
> - **forking** — old-style daemons that start, then fork a child process into the background, and the parent exits. systemd waits for the parent to exit, then tracks the child. `httpd` is a classic example.
> - **oneshot** — the process runs, does its job, and exits. systemd waits for it to finish before moving on. Good for startup scripts, not long-running daemons. Think of it like a task, not a service.


---

#### **`[Install]`**

`WantedBy=multi-user.target` — this is what `systemctl enable` uses. It creates a symlink so this service starts when the system reaches multi-user mode (basically normal boot).
