#permissions #files #advance 

**Sticky Bit** is a special permission used mainly on **directories** to control **who can delete files**.

Normally in Linux, if a directory has **write permission**, any user who can write to that directory can **delete or rename any file inside it**, even if the file belongs to another user.

The **sticky bit changes this behavior.**

When a directory has the sticky bit set:

- A user can **delete or rename only the files they own**
- The **directory owner** can delete files
- **root** can delete files
- Other users **cannot delete someone else's files**

This is used for **shared directories** where many users create files.

Example:  
The classic example is **`/tmp`**, where every user can create temporary files but should not delete other users’ files.

Typical permission of `/tmp`:

```
drwxrwxrwt
```

The **`t` at the end** shows the sticky bit.

Example to set sticky bit:

```bash
chmod +t shared_dir
```

Numeric form:

```
chmod 1777 shared_dir
```

- `1` → sticky bit
- `777` → everyone can read/write/enter the directory

Difference between **t and T**:

- `t` → sticky bit **and execute permission exists**
- `T` → sticky bit **but execute permission is missing**

So in simple words:  
**Sticky bit = users cannot delete other users' files inside a shared directory.**

Typical use cases:

- `/tmp`
- shared upload folders
- public collaboration directories

Short comparison with the other special bits:

- **SUID** → run program with **owner’s privileges**
- **SGID** → inherit **group ownership**
- **Sticky Bit** → **protect files from being deleted by others** in shared directories.
