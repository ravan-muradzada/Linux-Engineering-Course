## **What is SSH?**

SSH is an important and convenient solution that helps you establish remote connections to servers. It is also a dangerous solution. If your SSH server is visible directly from the Internet, you can be sure that sooner or later an intruder will try to connect to your server, intending to do harm.

The attacker uses the fact that SSH servers usually offer their services on port `22` and that still too many Linux servers have a root account that is allowed to log in over SSH. Based on that information, it is easy for an attacker to try to log in as root just by guessing the password.

---
## **How to make SSH more secure?**

We can take some measures to protect SSH servers against these kinds of attacks:

- Disable root login
- Disable password login
- Configure a nondefault port for SSH to listen on
- Allow specific users only to log in on SSH (not recommended)

## **SSH Config Files**

The main config file of SSH is `/etc/ssh/sshd_config` file. Moreover, there are other files for SSH config. These are under `/etc/ssh/sshd_config.d/` directory.  The reason the directory (and files under it) exists is clean separation. Instead of editing the main file directly (which can cause messy diffs or broken configs), you drop a small focused file that overrides just one setting.
`01-permitrootlogin.conf` specifically controls whether root can log in via SSH — on RHEL 9 it typically sets `PermitRootLogin no` by default for security.

Let's talk about some important fields in `./sshd_config` file:

- **`Port 22`** — the port sshd listens on. You can change this to a non-standard port (e.g. 2222) to reduce automated attacks, but for RHCSA just know the default is 22.
- **`AddressFamily any`** — accept both IPv4 and IPv6 connections. You can restrict to `inet` (IPv4 only) or `inet6`.
- **`ListenAddress 0.0.0.0`** and **`ListenAddress ::`** — listen on all network interfaces. `0.0.0.0` is all IPv4, `::` is all IPv6. If your server has multiple NICs, you could restrict SSH to one specific IP here.
- **`LoginGraceTime 2m`** — if a user connects but doesn't complete login within 2 minutes, the connection is dropped. Protects against hanging connections.
- **`PermitRootLogin prohibit-password`** — root can log in, but **only with SSH key**, not with a password. This is stricter than `yes` but less strict than `no`. We make sure if it is changed in `/etc/ssh/sshd_config.d/*.conf` file, too.
- **`StrictModes yes`** — sshd checks that key files and home directories have correct permissions before allowing login. Prevents accidents like world-writable `~/.ssh/`.
- **`MaxAuthTries 6`** — after 6 failed attempts, the connection is dropped. Limits brute force.
- **`MaxSessions 10`** — max number of open sessions per connection (relevant for multiplexing).
- **`PubkeyAuthentication yes`** — allows SSH key-based login. You definitely want this on (`yes`). It does not block log in with password, it is just a second chance.
- `PasswordAuthentication no` — blocks log in with password (if `no`).
- **`AllowUsers` / `AllowGroups` / `DenyUsers` / `DenyGroups`** — these are access control directives. They're blank here meaning no restriction is set. If you populate them, SSH checks in this order: `DenyUsers` → `AllowUsers` → `DenyGroups` → `AllowGroups`. Deny takes priority.

If we make any changes in that file, we need to [[07. Systemctl Command|restart]] the [[03. Systemd Services|service]]:

```bash
systemctl restart sshd
```


> [!Warning] Changing SSH Port
> If we change SSH port, Linux Firewall and SELinux does not accept it by default. We can disable them temporarily:
> ```bash
> systemctl stop firewalld.service
> setenforce 0
> ```

## **Disabling Password Login and Enable Public/Private Key Login**

**We generate a key pair** on your local machine with `ssh-keygen`. This creates two files — `~/.ssh/id_rsa` (private key, never leaves our machine) and `~/.ssh/id_rsa.pub` (public key, safe to share).

**We copy the public key to the server** using `ssh-copy-id user@server`.(This commands works for MobaXterm). This appends your public key into `~/.ssh/authorized_keys` on the server side. 

In CMD, we cannot use `ssh-copy-id user@server` command. Instead, we need to copy the public key and manually put it inside `~/.ssh/authorized_keys` file.

**When we connect**, this is what happens behind the scenes:

1. Our client says "I want to log in as this user"
2. The server looks in that user's `~/.ssh/authorized_keys` and finds our public key
3. The server generates a random piece of data, **encrypts it with our public key**, and sends it to us as a challenge
4. Our client **decrypts it using our private key** — only our private key can do this
5. Our client sends the decrypted result back to the server
6. The server checks — if it matches what it sent, it knows we have the correct private key
7. Connection is allowed

The key insight is that **our private key never travels over the network**. The server only ever sees our public key. The challenge-response proves we own the private key without ever revealing it. This is asymmetric cryptography in practice.
