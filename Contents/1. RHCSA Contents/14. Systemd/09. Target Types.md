## **Recap for Systemd, Daemons and Targets**

**systemd** is the first [[01. Introduction to Process|process]] that starts when your [[01. Introduction to Linux|Linux]] system boots — it gets PID 1, and everything else spawns from it. It's responsible for starting [[03. Systemd Services|services]], [[05. Mounting|mounting]] [[04. Filesystem|filesystems]], managing devices, and basically orchestrating the entire system. On [[02. Introduction to Red Hat|RHEL]], it fully replaced the old SysV init system.

**[[01. Introduction to Daemons and Systemd|Daemons]]** are background processes that run without a terminal — things like `sshd` (handles [[02. Configuring SSH|SSH]] connections), `httpd` ([[01. Introduction to Web and Application Servers|web server]]), `crond` (cron jobs). The "d" at the end of a process name usually tells you it's a daemon. systemd manages these daemons through **[[02. Units|unit]] files**.

**[[05. Systemd Targets|Targets]]** are systemd's grouping mechanism. A target bundles together a set of units that should be active at the same time. For example, `multi-user.target` means "bring up [[01. IP Address and Its Variations|networking]], login prompts, and standard services — but no graphical interface." `graphical.target` adds the desktop on top of that. You can think of targets as "system states."

## **Runlevels**

Runlevels come from the old **SysV init** system — before systemd existed. The idea was simple: define numbered "levels" that the system could be in, where each level activates a specific set of services. We might think each runlevel as a target from the standpoint of modern **systemd**.
We can run `runlevel` command, if prints the number(s) where the last number represents which runlevel currently we are.

We use `init` command to switch between runlevels. There are 7 runlevels:

- `init 0` ─ `poweroff.target`. When we run it, it shutdowns the system. The same with `poweroff` command.
- `init 1` ─ `rescue.target`. We are redirected to this runlevel if any serious problem is detected while booting. [[03. Crash in fstab and Debug|Problem in fstab]] can cause to go to this runlevel.
- `init 2` ─ `multiuser.target`. That is minimall installation target without network. 
- `init 3` ─ `multiuser.target`. That is main `multiuser.target` which also haves network access. That is real minimal installation.
- `init 4` ─ `reserve.target`. It is basically an environment to create custom target for ourselves.
- `init 5` ─ `graphical.target`. It is basically `multiuser.target` (runlevel 3) with Graphical User Interface.
- `init 6` ─ `reboot.target`. When we run it, it reboots (restarts) the system. The same with `reboot` command.

## **Modern Systemd Commands**

The commands above still work perfectly. However, their names has changed and they are [[07. Inode, Hard Link and Soft Link|soft links]] for modern ones. So, if we run this command: `ll /usr/sbin | grep -w "init"`, we will see this result:

```
lrwxrwxrwx. 1 root root  22 Aug 15  2025 init -> ../lib/systemd/systemd
```

The command which replaces `init` command is `systemctl isolate <TARGET_NAME>`. 
So, if we want to write:
- `init 0` - `systemctl isolate poweroff.target`
- `init 1` - `systemctl isolate rescue.target`
- `init 2` - It does not exist for now.
- `init 3` - `systemctl isolate multi-user.target`
- `init 4` - It does not exist for now.
- `init 5` - `systemctl isolate graphical.target`
- `init 6` - `systemctl isolate reboot.target`

This commands for the current environment, so they switch now, however they are not permanent. Because there is a default target in the system, and if we want to switch it permanently, we need to change it.

To see the current target, we need to run this command:

```bash
systemctl get-default
```

We can change the default target. If we do it, even after reboot the target will have been changed. The command for that:

```bash
systemctl set-default <TARGET_NAME>
# <TARGET_NAME> might be graphical.target or etc.
```

When we run this command, it removes the soft link for the previous target and creates a new one:
```
Removes /etc/systemd/system/<TARGET_NAME>
Created /etc/systemd/system/<TARGET_NAME> → /usr/lib/systemd/system/<TARGET_NAME>
```

## **More Info**

There was another mode name for `rescue.target` before systemd. We call it `single user mode`. They are similar, but not the exactly same.


> [!Info] `single user mod` and `rescue.target`
> **SysV single-user mode** was extremely bare — it mounted only the root filesystem (often read-only), started almost nothing, and dropped you straight into a root shell. No services, no network, no PAM authentication. Basically just enough to fix a broken system.
> 
> **systemd rescue.target** is more capable. It mounts all local filesystems, starts some basic systemd units, and still gives you a root shell — but it does prompt for the root password via PAM. Networking is still off, but the environment is more usable than old single-user mode.


**Systemd** add a new target (mode) which does not exist before systemd. That is called `emergency.target`. It doesn't map to any classic runlevel — it has no `runlevel*.target` symlink pointing to it. It's a purely systemd concept with no SysV equivalent. 

Emergency is closer to what old single-user mode actually was — root filesystem only (read-only), almost nothing running, no PAM in some configurations. It's what you use when even rescue mode can't come up properly.


> [!Info] Targets
> **emergency.target** → most minimal (closest to old single-user) → **rescue.target** → more usable → **multi-user.target** → full system
