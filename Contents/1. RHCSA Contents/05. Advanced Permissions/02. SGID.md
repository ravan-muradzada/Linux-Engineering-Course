#permissions #files #advance 

**SGID (Set Group ID)** is a special Linux permission that affects how **groups** work with files or directories.

For **files:**  
If a file has SGID set, when someone runs that file, the program runs with the **group permissions of the file’s group**, not the user’s current group. This is similar in idea to SUID, but it applies to **group identity instead of user identity**. In practice, SGID on files is not used very often on modern systems for security reasons.

For **directories (this is the common use):**  
When SGID is set on a directory, **all new files and subdirectories created inside it automatically inherit the directory’s group**, instead of the creator’s default group.

This is very useful for **team collaboration**. Imagine a shared directory for a project:

- Directory group: `developers`
- SGID enabled on that directory

Now when any user in that team creates a file inside it, the file will automatically belong to the **developers group**. This ensures everyone in the group can access and modify the shared files without constantly fixing group ownership.

Example:

```
chmod g+s project_dir
```

After this, new files inside `project_dir` inherit its group.

How SGID appears in permissions:

```
drwxrwsr-x
```

The **`s` in the group execute position** means SGID is active.

Difference between **`s` and `S`**:

- `s` → SGID is set **and** execute permission exists.
- `S` → SGID is set but **execute permission is missing** (usually a mistake).

Numeric form:

- SGID value = **2**

Example:

```
chmod 2775 project_dir
```

`2` → SGID  
`775` → normal permissions

So SGID mainly exists to make **shared directories behave correctly for groups**, which is why it is common in places like `/srv`, `/var/www`, or team project folders.

---

To implement SGID:

1. Create a directory. Change the group owner of this directory to `X_gr`.
2. Give `rwx` and then `s` to the group owner of this directory. `g+rwx` and `g+s`.
3. Create two users. Make the secondary group of each user `X_gr`.
4. Users can enter the directory and create files and they do not need to change the group permissions of their files. 
5. Users can modify each other's file. When `a_usr` tries to modify the file of `b_usr` (`file1`), the new file is created as `file1~`. Both the owner of `file1` and `file1~` is `a_usr`. `file1` is updated, `file1~` is previous one.
