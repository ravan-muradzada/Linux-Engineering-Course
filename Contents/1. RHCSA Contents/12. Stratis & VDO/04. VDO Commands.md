#vdo #commands #disk #vg #packages

Before start using [[03. Introduction to VDO|VDO]] commands, we need to have a [[01. Introduction to Disk|disk]] at least 5 GB and divide it to a [[02. Partitioning and Its Types|partition]] and then create a [[03. Volume Group|VG]] from it.

To be able to work with VDO, we need to download two packages beforehand:

```bash
dnf install vdo kmod-kvdo
```

Using VDO requires creating [[05. Logical Volume Management|LV]] as VDO type:

```bash
lvcreate -n <LV_NAME> --type vdo --size 9G --virtualsize 10T <VG_NAME>
```


Then we create filesystem:

```bash
mkfs.xfs -K <LV_PATH>
#OR:
mkfs.ext4 -E nodiscard <LV_NAME>
```


> [!Warning] `mkfs` options
> When you run `mkfs` on a VDO volume without these options, `mkfs` tries to **wipe/discard every block** on the volume to initialize it. On a regular disk this is fast. But VDO's virtual size can be huge (say 30GB virtual on 10GB physical), so `mkfs` would try to discard all 30GB — this takes a very long time and stresses VDO unnecessarily during format.
> - **`-K` on XFS** means "skip the discard step." XFS uses discard by default during `mkfs`, so `-K` just says don't bother.
> - **`-E nodiscard` on EXT4** does the same thing — the `-E` flag sets extended options, and `nodiscard` is telling mkfs.ext4 not to send discard/TRIM commands during format.
> 
Both options do the **same job on different filesystems** — skip the block discard phase during formatting.
VDO handles its own zeroing and deduplication internally, so the discard from `mkfs` is redundant anyway. You're not losing any safety by skipping it — you're just avoiding a slow, pointless operation on top of VDO.
Short version: without these flags, formatting a VDO volume can take many minutes or even hang. With them, it completes in seconds.



Then we can [[05. Mounting|mounting]] by adding this line to [[02. Syntax of fstab file|fstab]]:

```bash
UUID=<UUID>  <MOUNTPOINT>  <FILESYSTEM>  defaults 0 0
```
