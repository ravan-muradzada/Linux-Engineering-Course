## **What is Crontab?**

**crontab** is a scheduler in Linux. It lets us run commands automatically at specific times. 
Imagine we have to do some commands or scripts multiple times with a scheduler. Instead of running manually it each time, we set crontab and it automatically runs it.
*It comes from the **cron** service (a background [[01. Introduction to Process|daemon]]).*

Example idea:

- run a backup every day at 2 AM    
- clean logs every Sunday    
- restart a service every 5 minutes

***Each user has his own crontab.***

We can go https://crontab.guru/ website and write our crontab, then it will show us what our crontab means. This helps us to check our crontab whether it works properly or not.

---
## **Structure of Crontab**

```
* * * * * command
| | | | |
| | | | └── day of week (0-7)
| | | └──── month (1-12)
| | └────── day of month (1-31)
| └──────── hour (0-23)
└────────── minute (0-59)
```

This structure is saved in `/etc/crontab`. If we forget this structure, we just need to run `cat /etc/crontab` command and it will show us the structure above.

---

## **Example Crontabs**

- `* 11 * * *` - It runs in each day in each month in each minutes in `11`, starting from `11:00` to `11:59`.
- `0 11 * * 1-5` - It runs in each month, from monday to friday (weekdays) in `11:00`.
- `0 7-18 * * 1-5` - It runs on weekdays of each month, from `7'th` hour to `18'th` hour only `0'th` minute, starting with `7:00` to `18:00`.
- `0 */2 2 12 5` - It runs  every two hours on the hour (`0'th minute`) on `December 2` and `each Friday in December`.
- `00 1,11 * * *` - It runs each day of each month in `1:00` and `11:00`.


> [!Warning] Important Tip About Crontab
> In crontab config sometimes days may confuse us. As an example, we can see `18'th` day of any month and also `Friday` in this month. 
> 
> **What happens when `18'th` day is not `Friday`?**
> crontab uses **OR logic** when both a day-of-month and day-of-week are specified. So it actually runs on **two conditions**: 
> - `18'th` day of this month
> - each `Friday` of this month
> 
> The OR logic **only kicks in when both fields have specific values**. If either one is `*`, cron treats it as "not specified" and uses only the other field normally.
> 
>So `* * 20 01 *` runs only on January 20th — the `*` in day-of-week means it's ignored. And `* * * * 5` runs every Friday — the `*` in day-of-month means it's ignored. 
>
>The OR trap only happens when you put **actual values in both** day fields at the same time, like your earlier example `30 14 20 01 5`.
