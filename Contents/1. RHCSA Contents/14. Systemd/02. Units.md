## **What is Unit?**

In [[01. Introduction to Daemons and Systemd|systemd]], a **unit** is just a configuration file that tells systemd _what to manage_.

Daemons/services are only one type. systemd can manage other things too — that's why they're called "units" and not just "services."


> [!Info] Unit and Its Usage
> **A Unit** is actually instruction file to tell systemd what and how to do. It tells systemd how to run/handle any program. Inside a unit, we see which program to run (its path in the system), which user needs to run it, running it after which process and etc.
> The type tells systemd **how to handle it** — because managing a daemon is very different from managing a timer or a mount point.
> For example:
> - A `.service` — systemd needs to **start a process** and keep watching it
> - A `.timer` — systemd needs to **wait for a time**, then trigger something
> - A `.mount` — systemd needs to **attach a filesystem**, not run any process at all

Common unit types:

| Type       | What it manages                                                     |
| ---------- | ------------------------------------------------------------------- |
| `.service` | A daemon/program (most common)                                      |
| `.socket`  | A [[01. IP Address and Its Variations\|network]] port or IPC socket |
| `.timer`   | A scheduled task (like [[01. Introduction to Crontab\|cron]])       |
| `.mount`   | A filesystem [[05. Mounting\|mount]] point                          |
| `.target`  | A group of units (like a boot stage)                                |

The most important ones are `service` and `target`.
So when you run `systemctl start nginx`, systemd reads `nginx.service` — that's the unit file — and follows its instructions to start the daemon.

We can run `systemctl -t help` command to see all unit types.

## **Unit Locations**

The major benefit of working with Systemd is that it provides a uniform interface to start units. This interface is defined in the unit file. Unit files can occur in three locations:

- `/usr/lib/systemd/system`:  Contains default unit files that have been installed from RPM packages. You should never edit these files directly.
- `/etc/systemd/system`:  Contains custom unit files. It may also contain files that have been written by an administrator or generated by the systemctl edit command.
- `/run/systemd/system`:  Contains unit files that have been generated automatically.

If a unit file exists in more than one of these locations, units in the `/run` directory have highest precedence and will overwrite any settings that were defined elsewhere. Units in `/etc/systemd/system` have second highest precedence, and units in `/usr/lib/systemd/system` come last.
