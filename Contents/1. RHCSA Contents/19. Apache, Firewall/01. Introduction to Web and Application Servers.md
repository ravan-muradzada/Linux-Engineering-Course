#introduction #servers

## **What is Web Server?**

A **web server** handles static content — HTML files, images, CSS, JavaScript. When you open a browser and load a webpage, the web server just finds the file and sends it back. That's it. Examples: **Nginx, Apache**.

----
## **What is Application Server?**

An **application server** handles dynamic content — it runs your actual code, talks to databases, processes logic, and builds the response. When you log in to a website and it checks your password, that's the application server working. Examples: **Tomcat, Node.js, Gunicorn**.

---
## **Summary**

**In practice**, they usually work together:

```
Browser → Nginx (web server) → Gunicorn (app server) → Database
```

Nginx receives the request first. If it's a static file, it handles it alone. If it needs logic, it passes the request to the app server.

---
## **Load Balancer**

A load balancer sits in front of multiple servers and distributes incoming requests across them so no single server gets overwhelmed.

Imagine your app is running on 3 servers. Without a load balancer, all traffic hits one server while the other two sit idle. With a load balancer, requests get spread across all three.

**Why it matters:** if one server crashes, the load balancer stops sending traffic to it. Users don't notice anything. That's high availability.

**Distribution methods** (you'll see these terms):

- **Round robin** — request 1 goes to server 1, request 2 to server 2, and so on in rotation
- **Least connections** — sends to whichever server is least busy
- **IP hash** — same user always goes to same server (useful for sessions)

**Nginx vs HAProxy:**

Nginx is a web server + reverse proxy + load balancer in one. You use it when you need all those roles together. It's extremely common in modern stacks.

HAProxy is _only_ a load balancer/proxy — but it's older, battle-tested, and handles very high traffic with more advanced health checking and routing options. Big companies use it for critical infrastructure.
